ES6 与 webapck

2017-08-08

写js代码时,经常需要考虑浏览器的兼容性,而新的语法ES6,甚至ES7在旧版本浏览器中无法使用.于是诞生了babel,polyfill等,
那么如何和构建工具webpack结合使用呢.

babel 和 polyfill

babel是一个JavaScript编译器,借助转换器可以使用最新的语法,而不必等待浏览器支持.但是只支持新的语法,即所谓的语法糖,不支持新增的API,
如Promise全局对象.
polyfill是一种垫片,使用旧版本方法模拟出新的API,但不保证支持所有特性.
目前有两种使用方法,一种是添加到全局环境中和增加原型方法;另外一种是作为一个library使用,不会污染全局环境和修改原型

bable-loader

安装bable及loader:
npm install --save-dev babel-cli babel-core babel-preset-es2015 babel-loader
或者
yarn add --dev babel-cli babel-core babel-preset-es2015 babel-loader
其中babel-cli为命令行转码;
babel-preset-es2015为E2015转码规则,也可以按需安装其他的转码规则;
babel v6中已经babel API移到babel core中,所以安装babel-core而不是babel

在webpack中增加以下内容,

- `test`:规定需要转码的文件类型
- `include`:大多时候我们并不想转码第三方库的代码,因此使用`include`指定源码的文件夹,也可以使用`exclude`排除第三库所在的文件夹,支持正则表达.
- `loader`:需要注意到是`webpack`新版本中`loader`需为`*-loader`
- `cacheDirectory`:如果设置了该目录会将转换结果缓存起来,加快转码速度.可以设置为`true`可以,会使用默认路径`node_modules/.cache/babel-loader`,如果找不到`node_modules`,会使用系统默认临时文件目录

默认情况下babel并不会转码你的代码,可以在使用options指定转码规则,但是更推荐将该配置写到文件.babelrc中,打包过程中会自动读取该文件

babel-runtime

这里存在一个问题是,babel在每个文件中都插入了帮助函数,这样造成了代码的冗余.而babel-runtime的插件将这些帮助函数的代码转化为 require 语句,
比如自动引入babel-runtime/regenerator当你使用generators/async函数.
因此安装插件babel-plugin-transform-runtime作为开发依赖,而其本身依赖于babel-runtime
npm install --save babel-runtime
npm install --save-dev babel-plugin-transform-runtime
或者
yarn add babel-runtime
yarn add --dev babel-plugin-transform-runtime

可以为其指定参数 options
1.helpers,如果为 true,则将帮助函数插入文件中,若为 false,则指向moduleName,默认是true
2.polyfill,如果为 true则引入polyfill,作为工具函数不污染全局环境,默认为true
3.regenerator,默认是true,同样不会污染全局环境
4.moduleName,默认是babel-runtime

使用 polyfill

如果想要使用像Promise这种新增的全局对象,这时候就需要引入垫片polyfill,这里可以使用babel-polyfill.
不同的是babel转换是通过后台转换的,babel-polyfill是用于浏览器的.所以安装是:npm install --save babel-polyfill

实践后发现,完整的垫片文件很大,不符合要求.但是大多时候我们并不会使用到所有的新API,完全可以需要的时候才在文件中引用.
而引用的相同文件可以被webpack.optimize.CommonsChunkPlugin插件提取到公共文件中.

观察babel-polyfill的依赖于babel-runtime,core-js和regenerator-runtime.
一般情况下引用core-js即可,而core-js针对很多场景可以有不同的引入:

  • 浏览器引用完整编译 client
    • core.js 所有特性
    • shim.js 只包括标准代码
    • library.js 所有特性,但是不污染全局环境
  • 按需引入
    • fn 文件夹中指定内容
    • library 文件夹中指定内容,但是不污染全局环境
  • 按需引入一系列相关的特性,比如 es6/array,包括 ES6 Array 所有特性
  • 自定义 引入modules文件夹下,但是需要有足够的了解

开发及热部署

在最终的打包结果中,包含着不少webpack的代码,大多时候我们只想对比babel转换前后的代码,这个时候可以单独执行babel命令来查看转码后的结果

利用babel提供的监听功能可以不用敲命令实时查看转码代码,由于采用的增量跟新也并不会影响性能.
但是想更进一步,将命令加入到package.json的scripts中,使用npm run调用:
"babel": "nohup babel src -d dist/babel -w > /dev/null &"
其中nohup命令将任务调到后台执行,将结果输出到/dev/null等于丢弃到垃圾桶中.
这个时候产生了新的问题,如何查看后台是否在运行该程序以及关闭该程序,如果重复执行太多肯定不是我们想要的,手动ps命令查看进程号再kill掉就行.
原本想做一个防重入的功能,但是npm scripts中无法包含分号,又不想再增加一个shell脚本,因此做了以下工作:

  • babelpid 用来查看进程号,其中grep -v grep用来排除grep本身,awk命令用来只打印出需要的进程号
  • stopbabel 用来杀掉进程,可以批量删除.其中将npm --silent过滤掉npm本身的输出信息,xargs将前面的程序标准输出作为后面程序的参数
